# review

## 目的

対象Issueの承認済み実装方針と、現在のブランチにある変更差分を照合する。

最優先で確認するのは、次の2点である。

1. 人間が承認した実装範囲から逸脱していないか
2. 不要なファイル・処理・抽象化を追加していないか

あわせて、プロジェクト規約、Laravel・Bladeの実装品質、セキュリティ上の問題を確認し、根拠のある指摘だけを重大度別に報告する。

このSkillはコードを変更しない。

## 使用方法

実装完了後、Issue番号を指定して手動で実行する。

```text
/review 12
```

## 手順

1. 次のコマンドでIssue本文とコメントを取得する。

   ```bash
   gh issue view <Issue番号> --json number,title,body,comments,state,url
   ```

2. コメントから、最新の次の見出しを持つ方針シートを取得する。

   ```markdown
   ## 実装方針（承認済み）
   ```

   方針シートが見つからない場合は、レビューを行わず`BLOCKED`とする。

3. 次を確認する。

   ```bash
   git branch --show-current
   git status --short
   git diff --name-status HEAD
   git diff --stat HEAD
   git diff HEAD
   ```

   `git status --short`に未追跡ファイルがある場合は、対象ファイルを`Read`で確認する。

4. 方針シートのブランチ名と現在のブランチが一致しているか確認する。

5. 方針シートの「変更・追加」「変更なし（スコープ外）」と、実際の変更ファイルを照合する。

6. 変更差分を、次の順序でレビューする。
   1. 方針シートとの一致
   2. Issueの受け入れ条件との一致
   3. セキュリティ・データ保護
   4. プロジェクト規約との一致
   5. コード品質
   6. 不要コード・過剰設計の有無

7. 指摘を重大度別に整理して報告する。

## レビュー観点

### 方針シートとの一致

- 「変更・追加」に記載された範囲内か
- 「変更なし（スコープ外）」へ変更を加えていないか
- 方針シートにないファイルを追加・変更していないか
- 方針外の変更について、実装報告に理由が記載されているか
- 指定された既存コードを再利用しているか
- ブランチ名が方針シートと一致しているか

方針シートにない変更は、小規模であっても必ず指摘対象とする。

### Issueとの一致

- 受け入れ条件をコード上で満たしているか
- Issueにない機能を追加していないか
- 対象外とされた処理を実装していないか
- 仕様を推測して実装した形跡がないか

### セキュリティ・データ保護

Issueと変更内容に関連する範囲で、次を確認する。

- 入力値のバリデーション
- 認証・認可
- CSRF対策
- XSSにつながる未エスケープ出力
- Mass Assignment
- SQLインジェクション
- メール送信時の入力値処理
- 秘密情報や環境固有値のハードコード
- 個人情報の不要な保存・出力
- 破壊的または未承認のデータベース変更

関係のないセキュリティ項目を形式的に指摘しない。

### プロジェクト規約

次のルールと照合する。

- `.claude/rules/project-principles.md`
- `.claude/rules/laravel.md`
- `.claude/rules/blade-tailwind.md`
- `.claude/rules/issue-workflow.md`

ルール違反を指摘する場合は、根拠となるルールを明示する。

### コード品質

- 命名が既存コードと統一されているか
- Controllerなどへ責務が集中していないか
- 同じ処理を重複して実装していないか
- 既存の構成や実装パターンから不必要に外れていないか
- 到達不能コード、未使用コード、デバッグコードがないか
- エラー処理や境界値の考慮が明らかに不足していないか
- 変更理由を説明できないコードが含まれていないか

### 不要コード・過剰設計

- 受け入れ条件に不要なファイルを追加していないか
- 将来利用を想定した未使用コードを追加していないか
- 不要なService、Repository、Interface、Helperなどを追加していないか
- 1か所でしか使わない処理を無理に共通化していないか
- 既存コードで代替できる処理を新設していないか
- Issueと無関係なリファクタリングを混入させていないか

一般論だけを理由に、新しい抽象化やレイヤーの追加を提案しない。

## 重大度

### 🔴 重大

修正または人間の判断なしに後続工程へ進めない問題。

例:

- 方針シートのスコープ外変更
- 受け入れ条件を満たしていない
- セキュリティ上の脆弱性
- データ破壊や情報漏えいの可能性
- 未承認の依存パッケージや破壊的変更
- 実装方針と根本的に異なる設計

### 🟡 注意

直ちに重大事故にはつながらないが、品質・保守性・既存仕様への影響について人間の判断が必要な問題。

例:

- 命名や既存パターンとの不一致
- 責務の偏り
- 明確な重複
- エラー処理の不足
- 方針外だが影響の小さい変更

### 🔵 提案

対応しなくてもIssueの完了を妨げない任意の改善案。

提案は、具体的な利益があり、スコープを広げない場合に限る。将来を見越した抽象化や、好みだけに基づく提案は行わない。

## 判定

| 判定               | 条件                                                               |
| ------------------ | ------------------------------------------------------------------ |
| `PASSED`           | 🔴重大・🟡注意がない                                               |
| `NEEDS_DECISION`   | 🔴重大はないが、🟡注意がある                                       |
| `CHANGES_REQUIRED` | 🔴重大が1件以上ある                                                |
| `BLOCKED`          | 方針シート・差分・ブランチなど、レビューに必要な情報を確認できない |

🔵提案だけの場合は`PASSED`とする。

## 禁止事項

- コード、ローカルファイル、ブランチ、Issueを変更しない
- 指摘した問題を自動修正しない
- 他のSkillを自動実行しない
- テスト、Pint、Lint、ビルド、Migrationを実行しない
- コミット、Push、Pull Request作成を行わない
- 方針シートにない理想設計を新たな正解として押し付けない
- 根拠のない可能性や一般論を指摘として挙げない

`Bash`は、Issue・Git履歴・差分の読み取りに限って使用する。ファイルやGitの状態を変更するコマンドは禁止する。

## 出力形式

### 判定

`PASSED`、`NEEDS_DECISION`、`CHANGES_REQUIRED`、`BLOCKED`のいずれか1つ。

### 対象

- Issue番号・タイトル
- 現在のブランチ
- 変更ファイル

### 方針シートとの照合

- 方針どおりの変更
- 方針外の変更
- スコープ外への変更
- 未実装の受け入れ条件

該当しない項目は「なし」と記載する。

### 指摘事項

各指摘を次の形式で記載する。

```markdown
#### 🔴重大 / 🟡注意 / 🔵提案：指摘の要約

- 対象: `ファイルパス:行番号`
- 根拠: 方針シート、Issue、プロジェクト規約のいずれか
- 問題: 何が問題なのか
- 影響: 放置した場合に何が起きるか
- 対応方針: 修正内容ではなく、人間が判断すべき方向性
```

指摘がない重大度は省略する。

### 良好な点

方針シートへの適合、既存コードの再利用、適切な責務分離など、確認できた事実だけを簡潔に記載する。

### 次の工程

- `PASSED`：後続工程へ進める
- `NEEDS_DECISION`：🟡注意について人間の判断を待つ
- `CHANGES_REQUIRED`：自動修正せず、人間から修正指示を受ける
- `BLOCKED`：不足している情報または状態を解消して再実行する

## 完了条件

- 方針シート、Issue、変更差分、プロジェクト規約を照合している
- 指摘にファイル・行番号と根拠がある
- 重大度と判定が一致している
- コードやGitの状態を変更していない
