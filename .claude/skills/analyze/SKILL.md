# analyze

## 目的

対象のGitHub Issue、関連ドキュメント、既存コードを調査し、安全に実装へ着手できる状態か判定する。

変更予定のファイル、スコープ外、事前確認、動作確認を整理した実装方針案を人間へ提示する。

人間からGOサインを得た場合のみ、確定した実装方針を対象Issueのコメントへ記録する。

このSkillは、コードおよびローカルファイルを変更しない。

## 使用方法

Issue番号を指定し、実装前に手動で実行する。

```text
/analyze 12
```

## 手順

1. 次のコマンドでIssueを取得する。

   ```bash
   gh issue view <Issue番号> --json number,title,body,labels,comments,state,url
   ```

2. Issue本文とコメントから、次を整理する。
   - 目的
   - 背景
   - 受け入れ条件
   - 対象外
   - 関連するIssue・Pull Request

   人間による最新の明示的な回答や修正指示を有力な情報として扱う。

   Issue本文や関連仕様と矛盾する場合は、独断で判断せず確認事項として扱う。

3. `docs/`配下から関連する仕様を確認する。
   - `requirements/`
   - `features/`
   - `architecture/`
   - `decisions/`
   - `development/`

   個別の仕様書が存在しないことだけを理由に停止しない。

4. Issueに直接関係するRoute、Controller、FormRequest、Blade、Model、Mail、Test、設定ファイルなどを調査する。

   毎回リポジトリ全体を調査せず、参照関係や類似機能が見つかった場合のみ範囲を広げる。

5. 次の観点を確認する。
   - 再利用できる既存処理やコンポーネント
   - 同じ責務を持つ既存コード
   - 新規ファイルを作成する必要性
   - 共通処理や既存機能への影響
   - 破壊的変更の有無
   - 必要な事前確認と動作確認
   - Issueに関連するセキュリティ上の影響

6. 調査結果を判定し、実装方針案を人間へ提示する。

7. `READY`の場合は、GOサインまたは修正指示を待つ。

8. GOサインを得た場合のみ、承認された実装方針を対象Issueのコメントへ投稿する。

## 判定

| 判定                  | 意味                                                |
| --------------------- | --------------------------------------------------- |
| `READY`               | 具体的な実装方針を提示し、人間へ承認を求められる    |
| `NEEDS_CLARIFICATION` | 実装判断に必要な情報が不足または矛盾している        |
| `BLOCKED`             | 前提となるIssueやPull Requestが未完了で着手できない |

`READY`であっても、人間のGOサインを得るまではIssueコメントを投稿せず、後続工程へ進まない。

仕様が不足している場合も、質問を具体化するために必要な範囲でコード調査を続ける。

## 禁止事項

- コード、ローカルファイル、ブランチを変更しない
- 不明な仕様を推測しない
- 人間の承認前にIssueコメントを投稿しない
- 実装、コミット、Push、Pull Request作成を行わない

`Bash`は、情報取得と承認後のIssueコメント投稿に限って使用する。

## 出力形式

### 判定

`READY`、`NEEDS_CLARIFICATION`、`BLOCKED`のいずれか1つ。

### Issueの整理

- 目的
- 受け入れ条件
- 対象外

### 関連仕様

確認した`docs/`配下のファイルパスを記載する。

個別の仕様書が不要な場合は、次のように記載する。

```text
なし（Issue本文とコメントで判断可能）
```

### 現状調査・影響範囲

- 関連する既存コード
- 現在の挙動
- 再利用できる処理やコンポーネント
- 影響を受ける箇所
- 破壊的変更の有無
- 関連するセキュリティ上の懸念

### 実装方針案

`READY`の場合、次のテンプレートを使用する。

```markdown
## 実装方針

### 1. ブランチ構成

- `feat/#XXX-概要`

### 2. 実装ファイル

**変更・追加**

- `ファイルパス`
  - 変更内容
  - 再利用する既存コード
  - この変更が必要な理由

**変更なし（スコープ外）**

- 変更しないファイル・機能
- Issueの対象外となる処理

### 3. 事前確認

- [ ] 実装開始前に確認する前提条件

### 4. 動作確認

- [ ] 正常系で確認する内容
- [ ] 異常系で確認する内容
- [ ] 既存機能への影響確認
```

本当に必要なファイルだけを記載する。

未解決の仕様判断を「事前確認」へ残したまま`READY`にしない。

### 問題点・確認事項

不足・矛盾・判断が必要な事項を、可能な限り既存コードを根拠に具体化する。

確認事項がなければ「なし」と記載する。

### 次の工程

- `READY`：実装方針案へのGOサインまたは修正指示を求める
- `NEEDS_CLARIFICATION`：確認事項への回答を待つ
- `BLOCKED`：解消が必要な前提条件を示す

## 承認後の処理

人間からGOサインを得た場合、承認された実装方針案の見出しを次のように変更し、対象Issueへコメントする。

```markdown
## 実装方針（承認済み）
```

投稿前に、人間が承認した最終内容と一致していることを確認する。

同じ見出しを持つコメントがすでに存在する場合は、追加・上書きせず人間へ確認する。

投稿後、コメントURLまたは投稿完了を報告する。

## 完了条件

- 判定と必要な報告を提示している
- `READY`の場合、承認後に確定版をIssueコメントへ記録している
- コードおよびローカルファイルを変更していない
