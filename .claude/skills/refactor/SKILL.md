# refactor

## 目的

対象Issueで変更されたコードと`/review`の結果を確認し、挙動を変えずに改善できるリファクタリング候補だけを人間へ提案する。

評価基準は、一般論としての美しさではなく、次の2点とする。

1. 現在のコードに具体的な問題があるか
2. 既存のプロジェクトパターンまたはLaravel標準の責務へ、最小差分で揃えられるか

必要な提案がない場合は、無理に候補を作らず`NO_REFACTOR`とする。

このSkillはコードを変更しない。

## 使用方法

`/review`の完了後、Issue番号を指定して手動で実行する。

```text
/refactor 12
```

## 対象範囲

次の範囲だけを分析する。

- 現在のIssueで変更されたファイル
- 変更ファイルが直接利用する既存コード
- `/review`で指摘された箇所
- 人間が明示的に指定した箇所

リポジトリ全体を対象に、改善点を探索しない。

## 手順

1. 次のコマンドでIssue本文とコメントを取得する。

   ```bash
   gh issue view <Issue番号> --json number,title,body,comments,state,url
   ```

2. Issueコメントから、最新の次の方針シートを取得する。

   ```markdown
   ## 実装方針（承認済み）
   ```

3. 現在のブランチと変更差分を確認する。

   ```bash
   git branch --show-current
   git status --short
   git diff --name-status HEAD
   git diff HEAD
   ```

4. 現在の会話にある最新の`/review`結果を確認する。

   確認できない場合でも、方針シートと変更差分から分析可能であれば続行する。判断材料が不足する場合は`BLOCKED`とする。

5. 次の観点から、現在必要なリファクタリングだけを抽出する。
   - 長すぎるメソッドや深いネスト
   - 明確な重複処理
   - 不適切または不統一な命名
   - 未使用コードやデバッグコード
   - FormRequestの日本語属性定義など、既存規約からの逸脱
   - Controllerなどへの明確な責務集中
   - N+1や不要な重複クエリ
   - Laravel標準の責務へ委譲すべき処理
   - 方針シートにない不要なファイルや抽象化

6. 提案が必要な場合は、最大3件まで優先順位順に提示する。

## 提案条件

次のすべてを満たすものだけを提案する。

- 現在のコードに具体的な問題が存在する
- 今すぐ得られる明確な利益がある
- 外部から観測できる挙動を変更しない
- Issueの受け入れ条件を変更しない
- 現在のIssueと変更差分の範囲内で完結する
- 最小限の変更で対応できる
- 既存コードのパターンまたはLaravel標準の作法に沿う

次の理由だけに基づく提案は禁止する。

- 将来使うかもしれない
- 拡張性が高まる
- 再利用できそう
- 一般的に綺麗
- ベストプラクティスだから
- 疎結合になる
- テストしやすくなるかもしれない

## 新規抽象化の制限

原則として、次の新設を提案しない。

- Service
- Repository
- Interface
- Trait
- Helper
- Utility
- DTO
- 基底クラス
- 独自の共通レイヤー
- 新しいデザインパターン

例外として提案できるのは、次のすべてを満たす場合だけとする。

- 現在すでに具体的な重複または責務集中がある
- 同種の既存コードで同じ構成が採用されている
- 今回のIssue内で実際に利用される
- 新設しない方法では問題を解消できない
- 人間の承認後にのみ実施する

FormRequest、Mailable、Policy、Blade ComponentなどLaravel標準の責務も、必要性がある場合だけ提案する。

## 対象外

次はリファクタリングとして扱わず、人間へ差し戻す。

- 受け入れ条件を満たすための機能追加
- 表示結果、レスポンス、ルート、バリデーション結果の変更
- 認証・認可の仕様変更
- DBスキーマの変更
- セキュリティ上の不具合修正
- 新規パッケージの追加
- Issueの範囲外にある既存コードの整理

範囲外の問題は、必要であれば別Issue候補として報告するだけに留める。

## 判定

| 判定                 | 意味                                                 |
| -------------------- | ---------------------------------------------------- |
| `NO_REFACTOR`        | 現時点で必要なリファクタリングはない                 |
| `PROPOSAL_AVAILABLE` | 人間へ提示すべき具体的な候補がある                   |
| `BLOCKED`            | 方針シートや差分など、分析に必要な情報が不足している |

提案がないことは失敗ではない。

件数を埋めるための提案や、好みだけに基づく提案は行わない。

## 禁止事項

- コード、ローカルファイル、ブランチ、Issueを変更しない
- 提案を自動で実装しない
- 他のSkillを自動実行しない
- テスト、Pint、Lint、ビルド、Migrationを実行しない
- コミット、Push、Pull Request作成を行わない
- 方針シートにない理想設計を新しい正解として提案しない

`Bash`はIssue・ブランチ・差分の読み取りに限って使用する。

## 出力形式

### 判定

`NO_REFACTOR`、`PROPOSAL_AVAILABLE`、`BLOCKED`のいずれか1つ。

### 対象

- Issue番号・タイトル
- 現在のブランチ
- 分析した変更ファイル

### リファクタリング提案

`PROPOSAL_AVAILABLE`の場合、最大3件まで次の形式で記載する。

```markdown
#### 提案1：提案の要約

- 対象: `ファイルパス:行番号`
- 現在の問題:
- 最小限の修正案:
- 今得られる明確な利益:
- 挙動を維持できる根拠:
- 変更対象ファイル:
- 新規ファイル: なし / あり
- 既存パターンの根拠:
```

新規ファイルを提案する場合は、新設しない方法では解消できない理由も記載する。

### 提案しなかった事項

`/review`の指摘のうち、機能修正・仕様変更・範囲外などの理由でリファクタリング対象から除外したものを記載する。

該当しない場合は「なし」と記載する。

### 別Issue候補

現在のIssue外で発見した、対応を検討する価値がある問題だけを記載する。

該当しない場合は「なし」と記載する。

### 次の工程

- `NO_REFACTOR`：後続工程へ進める
- `PROPOSAL_AVAILABLE`：採用する提案を人間が指定するまで待つ
- `BLOCKED`：不足している情報を示し、解消後に再実行する

## 完了条件

- 現在のIssueと変更差分の範囲だけを分析している
- 各提案に現在の具体的な問題と利益がある
- 提案数が最大3件に収まっている
- 必要な提案がなければ`NO_REFACTOR`としている
- コードやGitの状態を変更していない
