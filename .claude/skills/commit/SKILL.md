# commit

## 目的

現在の変更内容とコミット履歴を確認し、次のルールでコミットする。

- 初回実装：変更したファイルを1ファイルずつ個別にコミットする
- レビュー・リファクタリング後の追加修正：未コミットの変更をまとめて1コミットする

コミットメッセージは、変更内容に適した接頭辞を付け、日本語で簡潔に記載する。

このSkillはコミットまでを担当し、PushやPull Request作成は行わない。

## 使用方法

コミットしたい変更がある状態で手動実行する。

text /commit

初回実装か追加修正かを、人間が引数で指定する必要はない。

## 手順

1. 現在のブランチと作業ツリーを確認する。

   bash git branch --show-current git status --short

   次の場合はコミットせず停止する。
   - 現在のブランチがmainまたはdevelop
   - 変更ファイルが存在しない
   - マージまたはリベースの競合が残っている

2. 変更内容を確認する。

   bash git diff --name-status git diff git diff --cached

   未追跡ファイルは内容を確認し、秘密情報・一時ファイル・生成物など、コミットすべきでないファイルが含まれていないか確認する。

3. 現在のブランチに、developから追加されたコミットがあるか確認する。

   bash git rev-list --count develop..HEAD

4. 結果に基づきコミット方式を決定する。
   - 0件：初回実装
   - 1件以上：レビュー・リファクタリング後の追加修正

5. 選択した方式に従ってステージとコミットを行う。

6. コミット後、作成したコミットと残っている未コミット変更を報告する。

## 初回実装

develop..HEADのコミット数が0の場合は、変更したすべてのファイルを1ファイルずつ個別にコミットする。

新規作成・既存ファイルの変更を問わず、1コミットにつき1ファイルとする。

各ファイルについて次の順序で処理する。

1. 対象ファイルの差分を確認する
2. 対象ファイルだけをステージする
3. ステージされた差分を確認する
4. 内容に合った日本語メッセージでコミットする
5. 次のファイルへ進む

bash git add -- '<ファイルパス>' git diff --cached git commit -m '<接頭辞>: <日本語の変更内容>'

git add .、git add -A、引数なしのgit addは使用しない。

ファイルのリネームについては、元ファイルと移動先ファイルを同じ変更単位として1コミットに含めてよい。

## 追加修正

develop..HEADのコミット数が1件以上の場合は、レビュー・リファクタリング後の追加修正として扱う。

すべての未コミット変更を確認し、関連する修正をまとめて1コミットにする。

対象ファイルは個別に指定してステージする。

bash git add -- '<ファイル1>' '<ファイル2>' '<ファイル3>' git diff --cached git commit -m '<接頭辞>: <日本語の変更内容>'

コミットメッセージは、まとめた修正全体を表す内容にする。

例:

text fix: レビュー指摘に基づきお問い合わせ処理を修正

text refactor: お問い合わせ処理の責務と条件分岐を整理

未コミット変更の中に、互いに無関係な作業が混在している場合は、自動的にまとめず停止して報告する。

## コミットメッセージ

次の形式を使用する。

text <接頭辞>: <日本語の変更内容>

主な接頭辞:

| 接頭辞   | 用途                               |
| -------- | ---------------------------------- |
| feat     | 新しい機能・画面・処理の追加       |
| fix      | 不具合やレビュー指摘の修正         |
| refactor | 挙動を変えないコード整理           |
| docs     | ドキュメントのみの変更             |
| style    | CSSや表示上の見た目の変更          |
| chore    | 設定、環境、CI、依存関係などの変更 |

例:

text feat: お問い合わせ画面を追加 feat: お問い合わせ用の入力検証を追加 fix: メールアドレスの検証処理を修正 refactor: お問い合わせ処理の条件分岐を整理 docs: Issue駆動開発の手順を追加 style: プロジェクト一覧の余白を調整 chore: CIワークフローを追加

変更内容に合わない接頭辞を機械的に使用しない。

メッセージは日本語で、対象ファイル名ではなく、そのファイルで行った変更を表す。

## 安全確認

各コミットの直前に、次を確認する。

- ステージ対象が意図したファイルだけである
- 秘密情報や.envが含まれていない
- ログ、一時ファイル、不要な生成物が含まれていない
- Issueと無関係な変更が含まれていない
- コミットメッセージが変更内容と一致している

不適切なファイルが見つかった場合は、そのファイルを削除・修正せず、コミットを停止して報告する。

## 禁止事項

- コードやファイルの内容を変更しない
- mainまたはdevelopへ直接コミットしない
- git add .やgit add -Aを使用しない
- --no-verifyを使用しない
- --amendを使用しない
- 既存コミットのrebase・reset・squashを行わない
- PushやPull Request作成を行わない
- コミット対象外の変更をstash・破棄・削除しない
- 空コミットを作成しない

## 出力形式

### 判定モード

- INITIAL
- FOLLOW_UP
- STOPPED

### 対象ブランチ

現在のブランチ名を記載する。

### 作成したコミット

各コミットについて次を記載する。

- コミットハッシュ
- コミットメッセージ
- 対象ファイル

### 未コミットの変更

コミット後に残っている変更を記載する。

残っていない場合は「なし」と記載する。

### 停止理由

STOPPEDの場合のみ、コミットを行わなかった理由を記載する。

### 次の工程

コミットが完了した場合は、PushまたはPull Request作成へ進めることを案内する。

## 完了条件

- 初回実装では、変更ファイルを1ファイルずつコミットしている
- 追加修正では、関連する未コミット変更を1コミットにまとめている
- すべてのコミットメッセージが接頭辞付きの日本語になっている
- コミット対象をファイルパスで明示してステージしている
- PushやPull Request作成を行っていない
