# issue

## 目的

指定された要件ドキュメントを、単独で実装・確認・マージできる単位へ分割する。
Issue案を人間へ提示し、GOサインを得た場合のみGitHub Issueを起票する。

※コード、ドキュメント、ローカルブランチの変更は一切行わない。

## 使用方法

```text
/issue docs/requirements/contact-form.md

```

## 手順

1. 指定された要件ドキュメントを読む。**実装範囲や受け入れ条件へ影響する未決事項がある場合は、推測でIssue化せず人間へ確認して停止する。**
2. 関連する既存Issueを確認し、重複を避ける。
3. 要件を実装可能なIssue単位へ分割する。
4. 各Issueのラベル、依存関係、実装順を整理する。
5. Issue案を人間へ提示し、GOサインまたは修正指示を求める。
6. GOサイン後、`gh label list` でラベルの存在を事前に確認する（存在しない場合は勝手に作成せず人間に報告する）。
7. 実装順（依存順）に従い、承認されたラベルを `--label` で必ず指定し起票する。

```bash
gh issue create \
  --title "<承認されたタイトル>" \
  --label "<承認されたラベル>" \
  --body-file -

```

- 前提Issueがある場合は、必ず親Issueを先に起票して「確定したIssue番号」を取得してから、後続Issueの本文へその番号を埋め込む。
- **起票失敗時のルール:** 起票に1件でも失敗した場合は後続Issueの起票を直ちに停止する。作成済みのIssueは削除せず、「作成済み一覧」「失敗したIssue案」「未作成のIssue案」を人間へ報告する。

8. 作成したIssue番号・URL・実装順を報告する。

## 分割原則

**【Issue数を最小限にする】**
分割条件を満たさない限り、Issueを増やさない。1件で安全に実装・確認できる内容は、複数Issueへ分割しない。

**【縦割りの機能単位と例外】**
DB・バックエンド・フロントエンドなど、技術レイヤーだけを基準にIssueを横分割しない。原則として、利用者が確認可能な価値を端から端まで実装できる「縦割りの機能単位」とする。
例外として、CI、開発環境、共通設定など、利用者向け機能に直接含められない技術基盤は、単独で完了条件を検証できる場合に限り `chore` として独立させてよい。

分割後、次の問いに「NO」となる場合は分割を見直すこと。

> 「各Issueは単独でマージされたとき、利用者または開発者が確認できる完了状態を持つか？」

悪い例（レイヤー分割）:

- Controllerを作成する
- Requestを作成する
- Bladeを作成する

良い例（機能分割）:

- お問い合わせ内容を入力できる
- 入力内容を検証して確認できる
- お問い合わせメールを送信できる

**【対象外と制約の適切な遺伝】**
要件ドキュメントの「対象外（スコープ外）」と「制約」は、各Issueに関係する項目を漏れなく転記する。要件全体に適用される事項は、すべてのIssueへ転記する。関係のない項目は機械的に複製しないこと。

## Issueの構成（テンプレート）

人間へ提示するIssue案、および `gh issue create` で起票する本文は、共通して次の構成とする。
※提示時のみ、先頭に「**ラベル:** `<ラベル名>`」「**実装順:** `<番号>`」を付与すること。

```markdown
## 目的

-

## 受け入れ条件

- [ ]

## 対象外

-

## 制約

-

## 依存関係

- 前提Issue: #XX （ない場合は「なし」）

## 関連要件

- `要件ドキュメントのパス`
```

**【受け入れ条件の記述ルール】**
可能な限り「前提・操作・期待結果」が分かる文章にし、人間またはAIがテスト手順として明確に判定できる状態にすること。

悪い例:

- [ ] お問い合わせ機能が正しく動く

良い例:

- [ ] お問い合わせ画面で必須項目を空のまま送信すると、各対象項目にエラーが表示される
- [ ] 有効な入力内容で送信すると、送信完了画面が表示される

## Issueのラベル

内容に応じて、以下の既存GitHubラベルから選択する。

| ラベル     | 用途                     |
| ---------- | ------------------------ |
| `feat`     | 新機能・画面・処理の追加 |
| `fix`      | 不具合修正               |
| `refactor` | 挙動を変えないコード整理 |
| `docs`     | ドキュメント変更         |
| `style`    | CSS・表示上の変更        |
| `chore`    | 設定・環境・CI・保守作業 |

## 禁止事項

- 人間の承認を得る前のIssue起票
- 要件ドキュメントにない機能や仕様の勝手な追加
- 不明点の推測、および技術レイヤー単位での横分割
- 既存Issueとの重複起票、GitHubラベルの新規作成
- コード・ドキュメント・ブランチの変更

`Bash` は既存Issue・ラベルの確認と、承認後のIssue起票だけに使用する。

## 完了報告

- 作成したIssue番号・タイトル・URL
- 各Issueのラベル
- 依存関係
- 推奨する実装順
- 次の工程の案内:

```text
実装順に従い、最初のIssueから /analyze <Issue番号> を実行してください。

```
