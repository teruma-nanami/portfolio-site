# Issue Workflow

このファイルは、Issue・ブランチ・コミット・Pull Requestの運用ルールを定める。

## 開発フロー

```text
/requirements
↓
必要に応じて /design-docs
↓
/issue
↓
/analyze
↓
/implement
↓
/commit
↓
/review
↓
必要に応じて /refactor
↓
Geminiレビュー
↓
修正がある場合は /commit
↓
/pr
↓
developへマージ
```

`/design-docs`は、Database・API・機能・アーキテクチャ・技術選定などの設計ドキュメントが必要な場合だけ実行する。

1回目の`/commit`では、初回実装を1ファイルずつコミットする。

レビュー・リファクタリング・Geminiレビュー後に変更が発生した場合のみ、2回目の`/commit`を実行し、追加修正をまとめてコミットする。

人間の承認が必要な工程を飛ばして進めない。

## Issue

- 実装作業はGitHub Issueを起点とする。
- Issueには目的・受け入れ条件・対象外・制約を記載する。
- 受け入れ条件は、完了したか明確に判定できる内容にする。
- Issueに記載されていない機能を追加しない。
- 範囲外の問題は、必要に応じて別Issueとして扱う。
- 依存するIssueがある場合は、本文に前提Issue番号を記載する。

## 実装方針

実装前に`/analyze`を実行し、人間が承認した実装方針をIssueコメントへ記録する。

```markdown
## 実装方針（承認済み）
```

`/implement`は、このコメントを実装範囲の正本として扱う。

承認済み実装方針が存在しない場合は、実装へ進まない。

## ブランチ

基準ブランチは`develop`とする。

Issue用ブランチは、最新の`develop`から作成する。

基本形式:

```text
<種類>/#<Issue番号>-<短いkebab-case>
```

例:

```text
feat/#12-contact-form
```

主な種類:

| 種類       | 用途                 |
| ---------- | -------------------- |
| `feat`     | 新機能               |
| `fix`      | 不具合修正           |
| `refactor` | 挙動を変えない整理   |
| `docs`     | ドキュメント変更     |
| `style`    | 表示・CSS変更        |
| `chore`    | 設定・環境・保守作業 |

ブランチ名には`#`などの特殊文字が含まれるため、Gitコマンドで使用する際は必ず引用符で囲む。

```bash
git switch -c "feat/#12-contact-form"
git switch "feat/#12-contact-form"
git push -u origin "feat/#12-contact-form"
```

シェルによる意図しない解釈を避けるため、引用符を省略しない。

ブランチ作成前に、次を確認する。

- 作業ツリーに未コミット変更がない
- `develop`が存在する
- `git pull --ff-only origin develop`で最新化できる
- 同名ブランチが存在しない

未コミット変更を勝手にstash・破棄・コミットしない。

## コミット

コミットメッセージは、接頭辞付きの日本語とする。

```text
feat: お問い合わせ画面を追加
fix: 入力値の検証処理を修正
refactor: 条件分岐を整理
```

### 初回実装

`/implement`の完了後、レビュー前に`/commit`を実行する。

`develop`からの追加コミットがない場合は、新規作成・既存変更を問わず、変更したファイルを1ファイルずつコミットする。

原則として、1ファイルにつき1コミットとする。

### 追加修正

`/review`、`/refactor`、Geminiレビュー後に変更が発生した場合は、再度`/commit`を実行する。

初回実装のコミットが存在する場合は、関連する未コミット変更をまとめて1コミットにする。

変更がない場合、2回目の`/commit`は実行しない。

無関係な変更が混在している場合は、自動的にまとめず停止する。

### 禁止事項

- `git add .`
- `git add -A`
- `--no-verify`
- `--amend`
- 既存コミットのrebase・reset・squash
- `main`または`develop`への直接コミット

## レビュー

初回実装をコミットした後に`/review`を実行する。

レビューでは、次を確認する。

- Issueと承認済み実装方針との一致
- スコープ外の変更
- 不要なファイルや抽象化
- プロジェクト規約との整合性
- セキュリティ上の問題
- コード品質

レビューはコードを自動修正しない。

必要な修正は、人間の判断後に行う。

## リファクタリング

`/refactor`は必要な場合のみ実行する。

- 挙動を変更しない
- 現在のIssueと変更差分の範囲に限定する
- 将来の拡張だけを理由に提案しない
- 人間が採用した提案だけを反映する

必要な改善がない場合は、変更しない。

## Pull Request

すべてのレビューと必要な追加コミットが完了した後、`/pr`を実行する。

作業ブランチをPushし、`develop`向けの通常Pull Requestを作成する。

```text
head: feat/#12-contact-form
base: develop
```

Draft PRは使用しない。

PR本文には、次を含める。

- Issueの概要
- 主な実装内容
- 承認済み方針との差異
- PR対象外
- 受け入れ条件
- 実施済み・未実施の確認内容
- `Closes #<Issue番号>`

実施していないテストや確認を完了済みとして記載しない。

## マージ

PRは人間が内容を確認した後、`develop`へマージする。

`develop`をGitHubのデフォルトブランチとし、PR本文の`Closes #<Issue番号>`によってマージ時にIssueを自動クローズする。

AIはPRを自動マージしない。

## 禁止事項

- Issueなしで実装を開始しない
- 承認済み実装方針なしで実装しない
- 初回コミット前にレビュー工程へ進まない
- Issueの範囲外を同じブランチへ混在させない
- 人間の確認なしに破壊的なGit操作を行わない
- `main`へ直接実装・コミット・PR作成を行わない
- PRを自動マージしない
