# Blade / Tailwind Rules

このファイルは、Blade・Tailwind CSS・画面側の実装ルールを定める。

## 基本方針

- 既存画面、既存コンポーネント、既存のTailwind記述を優先する。
- Issueの受け入れ条件に必要な最小限のUIだけを実装する。
- 将来の利用を想定したコンポーネントやデザインパターンを追加しない。
- 見た目だけでなく、操作性・レスポンシブ・アクセシビリティを確認する。

## Blade

- Bladeには表示と単純な条件分岐だけを記述する。
- DBアクセスや複雑な業務処理をBlade内で行わない。
- PHPコードを直接記述する`@php`は、既存パターンで必要な場合を除いて使用しない。
- URLの直書きを避け、名前付きRouteを使用する。
- フォームでは`@csrf`を必ず使用する。
- HTTPメソッドを偽装する場合は`@method`を使用する。
- 入力値の再表示には`old()`を使用する。
- バリデーションエラーは対象項目の近くへ表示する。

## レイアウト

- 共通レイアウトは既存のBladeレイアウトを再利用する。
- ページ固有の内容を共通レイアウトへ追加しない。
- ページごとに同じヘッダー、フッター、ナビゲーションを重複実装しない。
- 新しいレイアウトを作成する場合は、既存レイアウトで対応できない具体的な理由を確認する。

## Blade Component

- 複数箇所で利用されるUI、または独立した明確な責務を持つUIに使用する。
- 1回しか使わない小さなHTMLを、理由なくComponent化しない。
- 既存Componentで対応できる場合は新設しない。
- Componentへ業務処理やDBアクセスを持たせない。
- 属性やSlotを増やしすぎず、現在必要な入力だけを定義する。

## Tailwind CSS

- 原則としてTailwindのユーティリティクラスを使用する。
- 既存画面の余白、文字サイズ、角丸、色、幅のパターンに合わせる。
- 同じ見た目のために異なるクラス構成を乱立させない。
- 任意値は、既存のデザインや要件上必要な場合だけ使用する。
- 複雑なクラス列を理由なく共通化しない。
- カスタムCSSは、Tailwindだけでは適切に表現できない場合に限定する。
- `!important`は使用しない。

## レスポンシブ

- モバイルを基準に実装し、必要に応じて`sm`、`md`、`lg`などで拡張する。
- PC表示だけを基準に固定幅を設定しない。
- 横スクロール、文字切れ、ボタンの押しにくさが発生しないようにする。
- レスポンシブ対応のためだけに、同じ内容のHTMLを複製しない。
- 画面サイズによって操作できない機能を作らない。

## フォーム

- `label`と入力項目を適切に関連付ける。
- 必須項目は見た目とバリデーションの両方で判別できるようにする。
- エラー時も入力内容を保持する。
- 送信ボタンの文言は、実行される処理が分かる内容にする。
- フォーム送信中や二重送信への対策が必要な場合は、要件と実装方針に従う。
- placeholderだけをlabelの代わりに使用しない。

## アクセシビリティ

- クリック操作には用途に応じて`button`または`a`を使用する。
- クリック可能な`div`や`span`を安易に使用しない。
- 画像には内容に応じた`alt`を設定する。
- 装飾目的の画像やアイコンは、支援技術へ不要な情報を与えないようにする。
- フォーカス表示を理由なく削除しない。
- 色だけで状態やエラーを伝えない。
- 見出しレベルを不自然に飛ばさない。

## 出力とXSS

- ユーザー入力やDB由来の値は、原則として`{{ }}`で出力する。
- `{!! !!}`は、安全性が確認されたHTMLを意図的に表示する場合だけ使用する。
- ユーザー入力をそのままHTML属性やJavaScriptへ埋め込まない。
- URLや外部リンクは、想定された値か確認する。

## JavaScript

- Bladeだけで実現できる処理へ不要なJavaScriptを追加しない。
- JavaScriptが必要な場合は、ページ固有処理と共通処理を混在させない。
- Blade内へ長いスクリプトを直接記述しない。
- DOM要素の取得は、見た目のクラス名だけに依存させない。
- ライブラリやパッケージは、人間の承認なしに追加しない。
- JavaScriptが無効な場合に、主要な情報が閲覧不能にならないようにする。

## Figma・設計との整合性

- Figmaや画面設計がある場合は、文言、構成、状態、余白、色、レスポンシブを確認する。
- 設計との差異を独断で作らない。
- 実装上の制約で差異が必要な場合は、人間へ報告する。
- Figmaにない機能や状態を、便利そうという理由で追加しない。

## 禁止事項

- Blade内でのDBアクセスや複雑な業務処理
- 同じUIの重複実装
- 1回しか使わないUIの過剰なComponent化
- 要件にないアニメーションや装飾の追加
- 既存デザインと無関係な色・余白・サイズの導入
- ユーザー入力の未エスケープ出力
- アクセシビリティを損なう要素の使用
- Issueと無関係な画面修正
