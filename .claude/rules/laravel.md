# Laravel Rules

このファイルは、Laravelコードを実装・変更する際の共通ルールを定める。

## 基本方針

- Laravel標準の機能と既存コードのパターンを優先する。
- Issueの受け入れ条件に必要な最小限の実装にする。
- 将来の拡張だけを理由に、独自レイヤーや共通処理を追加しない。
- 同じ責務の既存コードがある場合は、新設せず再利用する。

## Controller

- ControllerはHTTPリクエストの受付と処理結果の返却に集中させる。
- バリデーション、複雑な業務処理、メール本文の組み立てをControllerへ直接記述しない。
- 単純な処理を無理に別クラスへ分離しない。
- Resource Controllerを使う場合は、Laravel標準のメソッド名に従う。
- 使用しないメソッドを持つControllerを作成しない。

## FormRequest

- 複数項目の入力検証や再利用される検証は、FormRequestへ分離する。
- 単純な入力が1項目だけの場合など、分離による利益がないときは過剰に作成しない。
- バリデーションルールは`rules()`へ記述する。
- 日本語の項目名は`attributes()`へ定義する。
- 独自メッセージが必要な場合だけ`messages()`を使用する。
- 認可が必要な場合は`authorize()`で明示する。

## Model

- Mass Assignmentを考慮し、`$fillable`または`$guarded`を明示する。
- リレーションはEloquentの標準メソッドを使用し、戻り値の型を記述する。
- DBアクセスを伴わない小さなドメイン処理は、責務が自然であればModelへ置いてよい。
- Controllerから同じクエリや処理が繰り返される場合は、既存パターンを確認して整理する。
- 1か所でしか使わない処理を、理由なくScopeや共通メソッドへ切り出さない。
- 不要なアクセサ、ミューテタ、イベント、Observerを追加しない。

## Route

- Routeは対応するControllerとHTTPメソッドが分かるように定義する。
- 名前付きRouteを使用し、URLの直書きを避ける。
- 複雑な処理をRouteクロージャへ記述しない。
- 認証・認可が必要なRouteには、適切なMiddlewareを設定する。
- 使用しないRouteや将来用のRouteを追加しない。

## 業務処理の分離

- Controllerが明確に肥大化する場合だけ、処理の分離を検討する。
- Laravel標準のFormRequest、Mailable、Policy、Jobなどで表現できる場合は、それらを優先する。
- Service、Repository、Interface、Action、DTOなどの独自レイヤーを自動的に追加しない。
- 独自クラスを追加する場合は、現在存在する具体的な責務または重複を解消できることを確認する。
- 新しい抽象化は、人間が承認した実装方針に含まれる場合だけ導入する。

## 認証・認可

- ログイン確認だけでなく、対象データを操作できる権限があるか確認する。
- 認可処理は、必要に応じてPolicy、Gate、Middlewareを使用する。
- ControllerやBlade上の表示制御だけを、認可の代わりにしない。
- 認証・認可の仕様を推測で追加または変更しない。

## Mail・Job

- メール送信にはMailableを使用する。
- メール本文は専用のBladeへ分離する。
- 宛先、送信元、秘密情報をコードへハードコードしない。
- 非同期処理は、現在の要件として必要な場合だけJob・Queueを使用する。
- 将来の負荷を想定しただけのQueue導入は行わない。

## Migration

- スキーマ変更は新しいMigrationファイルで行う。
- 既に共有・適用されたMigrationファイルを直接書き換えない。
- Migrationファイルの作成と実行を混同しない。
- Skillや人間の指示で禁止されている場合、`migrate`、`rollback`、SeederなどDB状態を変更するコマンドを実行しない。
- テーブル、カラム、外部キー、Indexなどの設計は`database.md`に従う。

## 設定・環境変数

- 環境ごとに変わる値や秘密情報は`.env`で管理する。
- アプリケーションコードから`env()`を直接呼ばず、設定ファイルを経由する。
- 新しい環境変数を追加した場合は、秘密値を含まない形で`.env.example`も更新する。
- 環境固有の値をコードへハードコードしない。

## 例外・ログ

- 例外を握りつぶさない。
- 利用者へ内部情報やスタックトレースを表示しない。
- ログへパスワード、トークン、個人情報を不用意に出力しない。
- `dd()`、`dump()`、`ray()`などのデバッグコードを残さない。
- 正常な業務上の分岐を、不要に例外として扱わない。

## セキュリティ

- 入力値はFormRequestなどで検証する。
- EloquentまたはQuery Builderを使用し、入力値をSQL文字列へ直接連結しない。
- Mass Assignment、認証・認可、CSRF、個人情報の扱いを確認する。
- ユーザー入力を安全であると仮定しない。
- セキュリティ上の判断が必要な場合は、独断で進めず人間へ確認する。

## 禁止事項

- Issueにない機能の追加
- 使用されないクラス・メソッド・設定の追加
- 一般論だけを理由とするService・Repository等の新設
- 既存パターンを無視した独自構成
- 秘密情報や環境固有値のハードコード
- 承認されていない破壊的変更
- 関係のないコードの同時修正
