# API Rules

このファイルは、APIの設計・実装・変更に関する共通ルールを定める。

## 基本方針

- 要件と受け入れ条件に必要なAPIだけを作成する。
- 既存APIのURL、命名、レスポンス形式を優先する。
- 将来利用する可能性だけを理由にEndpointや項目を追加しない。
- 不明な仕様を推測してRequest・Responseを確定しない。

## Endpoint

- URLはリソースを表す名詞を使用する。
- URLへ動詞を安易に含めない。
- 複数形・単数形は既存APIの規則に合わせる。
- ネストは必要最小限にし、深くしすぎない。
- 同じ目的のEndpointを重複して作成しない。
- バージョニングは要件として必要な場合だけ導入する。

例:

```text
GET    /api/projects
GET    /api/projects/{project}
POST   /api/projects
PUT    /api/projects/{project}
DELETE /api/projects/{project}
```

## HTTPメソッド

- `GET`：取得
- `POST`：新規作成
- `PUT`：リソース全体の更新
- `PATCH`：リソースの一部更新
- `DELETE`：削除

HTTPメソッドの意味と異なる処理を実装しない。

データを変更する処理に`GET`を使用しない。

## Request

- 入力値はFormRequestなどで検証する。
- 必須・任意・nullableを明確に区別する。
- クライアントから受け取る必要のない値を要求しない。
- 認証ユーザーやサーバー側で確定できる値を、クライアント入力へ依存させない。
- Mass Assignmentを考慮し、受け取った値を無条件に保存しない。
- 日付、数値、真偽値、列挙値の形式を明確にする。

## Response

- 既存APIのJSON形式へ合わせる。
- 同じ種類のレスポンスで構造を変えない。
- 必要なデータだけを返す。
- DBの内部構造や不要な個人情報をそのまま公開しない。
- Eloquent Modelを無条件に直接返さず、必要に応じてResourceを使用する。
- 成功・失敗を独自の曖昧な文字列だけで表現しない。

例:

```json
{
  "data": {
    "id": 1,
    "name": "Portfolio"
  }
}
```

一覧にページネーションが必要な場合は、Laravel標準の形式または既存形式を使用する。

## HTTPステータス

主なステータスコードは次を基準とする。

| ステータス | 用途                 |
| ---------- | -------------------- |
| `200`      | 取得・更新の成功     |
| `201`      | 新規作成の成功       |
| `204`      | 本文を返さない成功   |
| `400`      | 不正なリクエスト     |
| `401`      | 未認証               |
| `403`      | 権限不足             |
| `404`      | 対象が存在しない     |
| `409`      | 競合                 |
| `422`      | バリデーションエラー |
| `500`      | サーバー内部エラー   |

成功・失敗に関係なく常に`200`を返す設計は行わない。

## バリデーションエラー

- バリデーションエラーは`422`で返す。
- エラー対象と内容をクライアントが判別できる形式にする。
- 同じプロジェクト内でエラー形式を統一する。
- 内部実装やSQLなど、利用者に不要な情報を含めない。

Laravel標準形式を変更する場合は、人間が承認した仕様に従う。

## 認証・認可

- 認証が必要なEndpointはMiddlewareで保護する。
- 認証済みであることと、対象リソースを操作できることを区別する。
- リソース単位の認可にはPolicyやGateを使用する。
- URLやRequest内のユーザーIDだけを信用しない。
- Blade側やフロント側の表示制御だけを認可の代わりにしない。
- 認証方式を独断で追加・変更しない。

## セキュリティ

- ユーザー入力を信用しない。
- SQL、XSS、Mass Assignment、IDORなどの影響を確認する。
- 秘密情報、トークン、内部エラー、スタックトレースをResponseへ含めない。
- 個人情報は必要な項目だけ返す。
- 外部入力をURL、ファイルパス、コマンドへ直接使用しない。
- Rate Limitが必要な公開Endpointでは、既存方針または要件を確認する。

## Controller・Resource

- ControllerはRequestの受付、処理の呼び出し、Responseの返却に集中させる。
- 複雑な業務処理をControllerへ直接記述しない。
- JSON整形が複雑または再利用される場合はResourceを使用する。
- 単純なResponseまで形式的にResource化しない。
- 使用しないResource、Service、Repositoryを追加しない。

## 一覧・検索

- 一覧取得は必要に応じてページネーションする。
- 件数制限なしの大量取得を行わない。
- 検索・並び替え・絞り込み条件は、許可された項目だけ受け付ける。
- クライアントから受け取ったカラム名を無条件に`orderBy`などへ渡さない。
- N+1が発生しないよう、必要な関連だけを取得する。

## 更新・削除

- 更新対象が存在しない場合は適切に`404`を返す。
- 更新可能な項目を明確に制限する。
- 削除時は関連データへの影響を確認する。
- 冪等性が必要な処理では、同じRequestを複数回送った場合の挙動を明確にする。
- 二重送信や重複登録を防ぐ必要がある場合は、要件と設計方針に従う。

## 外部API

- 外部APIのURL、キー、秘密情報は設定ファイルと環境変数で管理する。
- タイムアウトを設定する。
- 通信失敗、認証失敗、Rate Limit、想定外Responseを考慮する。
- 外部APIのResponseを無検証で利用しない。
- リトライやキャッシュは、現在の要件として必要な場合だけ導入する。
- 外部サービス固有の処理をアプリケーション全体へ広げない。

## ドキュメントとの整合性

- `docs/api/`に設計書がある場合は、その仕様を正とする。
- Endpoint、Request、Response、認証、エラー形式を設計書と一致させる。
- 仕様変更が必要な場合は、実装だけを先に変更しない。
- OpenAPIなどを採用する場合は、実装と定義を同期する。

## 禁止事項

- 要件にないEndpointやResponse項目の追加
- HTTPメソッドと処理内容の不一致
- 常に`200`を返す独自エラー処理
- 認可を伴わない他ユーザーリソースへのアクセス
- Eloquent Modelや内部情報の無制限な公開
- 秘密情報・個人情報・スタックトレースの返却
- 将来利用だけを理由とするバージョニングや共通化
- 承認されていない外部API・パッケージの追加
